# Team Environment Movement V1

## Purpose
`team_environment_movement_v1` is the **team-state-only** successor to
`team_environment_movement_v0`. It describes directional change in a team's observed NFL
environment over deterministic early/late windows, and — unlike v0 — it carries **no fantasy-point
fields**, keeping the movement artifact inside the TTS v1 boundary (observed NFL team state, not
fantasy outputs).

This is the producer-side resolution of issue #34. v0 remains unchanged for the current
TIBER-Fantasy consumer; v1 is the cleaned boundary. See "Migration" below.

## Artifact
- Output path: `output/team_environment_movement_v1.json`
- Generated by `buildTeamEnvironmentMovementV1` during `npm run pipeline`.
- Not committed to git (ignored by the `output/` policy; it has no downstream consumer yet — see
  `docs/output-artifact-policy.md`). Regenerate locally from the committed demo fixture
  `data/fixtures/team_week_raw/team_week_raw_v0.movement_demo.sample.json`.

## Shape
- `artifact`: `team_environment_movement_v1`
- `generatedAt`: ISO timestamp
- `metadata`: provenance + coverage metadata (shared structurally with v0)
- `teams`: per team-season movement entries

### Window averages (`earlyWindow.averages`, `lateWindow.averages`)
Team-state fields only:
- `pointsPerDrive`
- `epaPerPlay`
- `successRate`
- `explosivePlayRate`
- `pressureRateAllowed`
- `secondsPerPlay`
- `neutralPassRate`
- `volatilityScore` (a Teamstate stability score, not a fantasy output; drives volatility direction)

### Deltas (`deltas`)
Same team-state fields as the averages, **excluding** `volatilityScore`.

### Movement labels (`movement`)
Identical semantics and thresholds to v0:
- `offenseDirection`, `passEnvironmentDirection`, `paceDirection`, `pressureDirection`,
  `volatilityDirection`, and a readable `verdict`.

## Difference from v0
v1 omits the four legacy fantasy-point fields from window averages and deltas:
- `fantasyPointsForQB`
- `fantasyPointsForRB`
- `fantasyPointsForWR`
- `fantasyPointsForTE`

**Movement labels are unchanged.** In v0 these fields were inert reported metrics — no movement
direction or verdict was ever derived from them. v1 is built as the team-state projection of the
same v0 computation (`buildTeamEnvironmentMovementV1` calls `buildTeamEnvironmentMovementV0` and
strips the fantasy fields), so directional output is provably identical.

## Migration (TIBER-Fantasy → v1) — next PR, not done here
The live TIBER-Fantasy movement consumer today:
- hard-requires the `artifact` literal `team_environment_movement_v0` (fails closed otherwise), and
- reads window/deltas via a tolerant pass-through (`asRecordOrNull`) and **never references the
  fantasy-point fields** (verified: zero `fantasyPointsFor*` references anywhere in TIBER-Fantasy).

Because Fantasy never reads the fantasy-point fields, moving it to v1 is functionally safe; it
requires only a Fantasy-side change to accept the `team_environment_movement_v1` literal and point
`TEAM_ENVIRONMENT_MOVEMENT_ARTIFACT_PATH` at the v1 artifact. That change is a separate PR in the
TIBER-Fantasy repo (out of scope for this producer-side Teamstate PR), after which the committed
representative fixture can move from v0 to v1 under the `output/` policy.

Until that migration lands, v0 stays the committed/consumed artifact and retains its legacy
fantasy-point fields as documented v0 boundary debt.
