# Team Environment Movement V1

## Purpose
`team_environment_movement_v1` is the **team-state-only** successor to
`team_environment_movement_v0`. It describes directional change in a team's observed NFL
environment over deterministic early/late windows, and — unlike v0 — it carries **no fantasy-point
fields**, keeping the movement artifact inside the TTS v1 boundary (observed NFL team state, not
fantasy outputs).

This is the producer-side resolution of issue #34. v0 remains unchanged for the current
TIBER-Fantasy consumer; v1 is the cleaned boundary. See "Migration" below.

## Artifact
- Output path: `output/team_environment_movement_v1.json`
- Generated by `buildTeamEnvironmentMovementV1` during `npm run pipeline`.
- **Committed** as the single representative `fixture_scaffold` fixture under `output/` (tracked via
  the `output/` policy allow-list; it now backs the live TIBER-Fantasy consumer — see
  `docs/output-artifact-policy.md`). It is reproducible from the committed demo fixture
  `data/fixtures/team_week_raw/team_week_raw_v0.movement_demo.sample.json`; only `generatedAt`
  changes between runs. To regenerate only this file without rewriting the rest of `output/`:

  ```bash
  npm run build
  node dist/src/pipeline/runTeamStatePipeline.js \
    --input data/fixtures/team_week_raw/team_week_raw_v0.movement_demo.sample.json \
    --output /tmp/tts-movement
  cp /tmp/tts-movement/team_environment_movement_v1.json output/team_environment_movement_v1.json
  ```

## Shape
- `artifact`: `team_environment_movement_v1`
- `generatedAt`: ISO timestamp
- `metadata`: provenance + coverage metadata (shared structurally with v0)
- `governance`: explicit, producer-owned governance metadata (see "Governance metadata" below)
- `teams`: per team-season movement entries

## Governance metadata (`governance`)
Producer-side resolution of issue #40. This block lets a downstream consumer distinguish a governed
promoted production artifact from a fixture/scaffold, an ungoverned/local artifact, or an
unknown/missing-governance artifact — **without inferring trust from a path**. A `/promoted/`
location is at most a weak hint; it is never the sole governance proof.

It is additive: the existing `metadata`, window averages, deltas, and movement labels are unchanged.
Only a new top-level `governance` object is appended to the artifact.

### Fields
- `governanceStatus`: one of
  - `governed` — a governed, promoted production artifact.
  - `fixture` — a fixture/scaffold/sample artifact (demo data, not production truth).
  - `ungoverned` — a real-but-ungoverned artifact (e.g. partial/local/dev output).
  - `unknown` — governance could not be established.
- `governanceSource`: how the status was established —
  - `explicit_marker` — the producer supplied a recognized provenance/governance marker for the run.
  - `path_inference` — status was inferred from the input path only (a weak hint, not proof).
  - `unknown` — no basis for a governance determination was available.
- `contractVersion`: the dataset-level contract literal, always exactly `team_environment_movement_v1`.
- `generatedAt`: dataset-level timestamp, mirrors the artifact's top-level `generatedAt`.
- `promotedAt` (optional): a promotion timestamp, only present when distinct and meaningful.
- `promotionNotes` (optional): non-advisory provenance notes. Present when the status was inferred
  from the path, recording that the path (e.g. `/promoted/`) is only a weak hint.

### How status and source are derived
The status is the explicit projection of the resolved dataset `metadata.provenanceStatus`:

| `metadata.provenanceStatus` | `governanceStatus` |
| --- | --- |
| `governed_real_data` | `governed` |
| `fixture_scaffold`, `sample` | `fixture` |
| `partial_real_data` | `ungoverned` |
| `unknown_provenance` | `unknown` |

`governanceSource` is resolved in three ways:
- `explicit_marker` — the producer supplied a recognized provenance marker for the run.
- `path_inference` — no marker, but the input path yields a usable provenance signal (e.g. a
  `data/sample/` or `fixtures/` path); a `promotionNotes` entry records that the path is only a weak
  hint.
- `unknown` — no marker and the path yields no usable signal (status resolves to `unknown`); a
  `promotionNotes` entry records that governance could not be established. This is kept distinct from
  `path_inference` so consumers can tell "no basis at all" from "inferred from a weak path".

Because `governed_real_data` only resolves to a governed status when coverage is full-league, a
`/promoted/` path can never, on its own, make an artifact `governed` — a bare `/promoted/` path is
unrecognized provenance, so on its own it yields `unknown`/`unknown`.

> Scope note: this is provenance/governance metadata only. It adds **no** fantasy scoring, point
> prediction, rankings, dynasty, recommendation/start-sit/trade/waiver semantics, Team Direction
> semantics, or any consumer-side activation. Downstream consumption is intentionally out of scope.

### Window averages (`earlyWindow.averages`, `lateWindow.averages`)
Team-state fields only:
- `pointsPerDrive`
- `epaPerPlay`
- `successRate`
- `explosivePlayRate`
- `pressureRateAllowed`
- `secondsPerPlay`
- `neutralPassRate`
- `volatilityScore` (a Teamstate stability score, not a fantasy output; drives volatility direction)

### Deltas (`deltas`)
Same team-state fields as the averages, **excluding** `volatilityScore`.

### Movement labels (`movement`)
Identical semantics and thresholds to v0:
- `offenseDirection`, `passEnvironmentDirection`, `paceDirection`, `pressureDirection`,
  `volatilityDirection`, and a readable `verdict`.

## Difference from v0
v1 omits the four legacy fantasy-point fields from window averages and deltas:
- `fantasyPointsForQB`
- `fantasyPointsForRB`
- `fantasyPointsForWR`
- `fantasyPointsForTE`

**Movement labels are unchanged.** In v0 these fields were inert reported metrics — no movement
direction or verdict was ever derived from them. v1 is built as the team-state projection of the
same v0 computation (`buildTeamEnvironmentMovementV1` calls `buildTeamEnvironmentMovementV0` and
strips the fantasy fields), so directional output is provably identical.

## Migration (TIBER-Fantasy → v1) — done
The migration is complete across both repos:

- **TIBER-Fantasy (PR #225, merged):** the movement consumer now accepts both the
  `team_environment_movement_v1` and legacy `team_environment_movement_v0` literals (fails closed on
  anything else), and its default `TEAM_ENVIRONMENT_MOVEMENT_ARTIFACT_PATH` resolution prefers the
  v1 artifact when present, falling back to v0 only for transition safety. It never referenced the
  fantasy-point fields, so the move is behavior-neutral.
- **TIBER-Teamstate (this lane):** the committed representative `output/` fixture is now
  `team_environment_movement_v1.json` (this artifact), and `team_environment_movement_v0.json` is no
  longer tracked. See `docs/output-artifact-policy.md`.

Because v1 is now the committed fixture, Fantasy's default resolves v1 first and the v0 fallback is
dormant. Removing Fantasy's v0 acceptance is a later, separately-scoped cleanup and is intentionally
**not** bundled here.
